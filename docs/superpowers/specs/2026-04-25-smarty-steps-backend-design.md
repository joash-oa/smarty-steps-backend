# Smarty Steps Backend — Design Spec

## Overview

Backend for Smarty Steps, a learning app for 5–8 year olds covering Math, Science, and English. Parents create accounts and manage learner profiles. Learners complete lessons and exercises, earn stars and XP, level up, and appear on leaderboards. Content is seeded from the NY State curriculum via Common Standards Project / ASN APIs, with lesson content generated by Claude (Anthropic API).

---

## Architecture

**3-layer architecture — strictly enforced:**

```
API (routers) → Service (business logic) → DAO (SQLAlchemy queries)
```

- Routers: HTTP only — parse input, call service, return response. No logic.
- Services: All business rules. No FastAPI imports. No raw SQL.
- DAOs: SQLAlchemy queries only. No business logic.

**Project structure:**
```
app/
  api/          # FastAPI routers
  services/     # Business logic
  daos/         # Data Access Objects — sole interface to PostgreSQL
  db/           # SQLAlchemy ORM models (entities) and DB session
  schemas/      # Pydantic request/response schemas
  clients/      # External service clients (Cognito, Claude API)
  core/         # Config, settings
tests/
  api/          # httpx.AsyncClient integration tests
  services/     # Unit tests (mock DAOs)
  daos/         # Integration tests (real test DB)
```

**Stack:**
- Python + FastAPI
- PostgreSQL (Amazon RDS) with JSONB for lesson content
- SQLAlchemy + Alembic
- AWS Cognito (auth — parent registration, login, JWT issuance)
- Kubernetes (compute)
- Claude (Anthropic API) — lesson and chapter quiz content generation

## Database Schema

All primary keys use `uuid7()` (time-ordered UUIDs via `pg_uuidv7` extension) for index locality.

|--------|------|-------------|-----------|
| `id` | UUID | PK, DEFAULT uuid7() | Time-ordered for index performance |
| `cognito_id` | VARCHAR | UNIQUE NOT NULL | Foreign reference to Cognito user pool — used to look up parent on every authenticated request |
| `email` | VARCHAR | UNIQUE NOT NULL | Stored for display; Cognito is auth source of truth |
| `pin_hash` | VARCHAR | NOT NULL | bcrypt hash of 4-digit PIN; used for parent dashboard access separate from main login |

### `lessons`

| Column | Type | Constraints | Reasoning |
|--------|------|-------------|-----------|
| `id` | UUID | PK, DEFAULT uuid7() | |
| `chapter_id` | UUID | FK → chapters(id) ON DELETE CASCADE | Lesson belongs to one chapter; cascade removes lessons if chapter is deleted |
| `standard_id` | UUID | FK → standards(id) ON DELETE SET NULL | Lesson generated from this standard; SET NULL preserves lesson if standard is removed |
| `subject` | VARCHAR | NOT NULL | Denormalised from chapter for efficient subject-level queries without join |
| `title` | VARCHAR | NOT NULL | Human-readable lesson title |
| `difficulty` | VARCHAR | NOT NULL DEFAULT 'easy' | `easy \| medium \| hard`; lessons within a chapter progress easy → medium → hard by `order_index` |

| `content` | JSONB | NOT NULL | Full lesson content including correct answers — never sent raw to client |
| `stars_available` | INT | DEFAULT 3 | Max stars earnable; always 3 for lessons |

Lessons contain mixed exercise types (multiple_choice, fill_blank, matching) within the JSONB — no single `exercise_type` column.

**Sanitized JSONB sent to client (correct answers stripped):**

- `correct_word` (fill_blank) — stripped entirely
- `pairs` (matching) — stripped and replaced with shuffled `left_items: [string]` and `right_items: [string]` arrays so the client can display items without knowing the correct pairings
- `explanation` — stripped; revealed only via `check-answer` response

## Indexes

| Index | Table | Columns | Reasoning |
|-------|-------|---------|-----------|
| `idx_learners_parent_id` | `learners` | `parent_id` | Every authenticated request lists a parent's learners |
| `idx_chapters_subject_order` | `chapters` | `(subject, order_index)` | Curriculum fetch always filters by subject and orders by index |
| `idx_lessons_chapter_order` | `lessons` | `(chapter_id, order_index)` | Curriculum fetch fetches all lessons for a chapter in order |
| `idx_lessons_standard` | `lessons` | `standard_id` | Content generation checks which lessons already exist per standard |
| `idx_lesson_progress_learner` | `lesson_progress` | `learner_id` | Lock state computation queries all progress for a learner; UNIQUE covers `(learner_id, lesson_id)` but single-column index needed for `WHERE learner_id = ?` alone |
| `idx_learners_total_stars` | `learners` | `total_stars` | Leaderboard all-time ranking sorts by total_stars |
| `idx_learners_last_active` | `learners` | `last_active_at` | Weekly/monthly leaderboard filters by last_active_at |
| `idx_standards_subject_grade` | `standards` | `(subject, grade_level)` | Standards sync looks up existing records by subject and grade |

`UNIQUE` constraints on `lesson_progress(learner_id, lesson_id)` and `chapter_quizzes(learner_id, chapter_id)` create indexes automatically — no explicit index needed for those columns.

## Business Rules

### Star Calculation (per lesson attempt)
| Accuracy | Stars |
|----------|-------|
| 100%     | 3     |
| 70–99%   | 2     |
| 50–69%   | 1     |
| < 50%    | 0     |

### XP Calculation (lessons)
```
xp_earned = base_xp × (stars_earned + 1)
base_xp = 10
```
0 stars = 10 XP, 1 star = 20 XP, 2 stars = 30 XP, 3 stars = 40 XP.

### Chapter Quiz XP (star multiplier)
```
effective_stars = stars_earned × multiplier

Multiplier tiers:
  3 stars → ×2   → 6 effective stars → xp = 10 × (6 + 1) = 70
  2 stars → ×1.5 → 3 effective stars → xp = 10 × (3 + 1) = 40
  1 star  → ×1   → 1 effective star  → xp = 10 × (1 + 1) = 20
  0 stars → ×1   → 0 effective stars → xp = 10 × (0 + 1) = 10

total_stars += effective_stars
```

`chapter_quizzes.stars_earned` stores raw stars (0–3). Effective stars are computed at submission time.

### Level Thresholds
Linear: `level = floor(xp / 100) + 1` (no cap)
- Level 1: 0–99 XP
- Level 2: 100–199 XP
- Level 3: 200–299 XP (and so on)

`ProgressService` recalculates and updates `learners.level` after every XP change.

### Replay Rules
- Lessons and chapter quizzes are always replayable.
- `lesson_progress` upserted: only update `stars_earned`, `score_correct`, `score_total`, `time_seconds` if new attempt earns more stars than previous best.
- `chapter_quizzes` upserted on replay: only update if new `effective_stars` > previous best.
- Learner `total_stars` and `xp` incremented by delta (new − previous best). Negative delta → no update.
- Same quiz content served on chapter quiz replay — no regeneration.

### Lesson Locking
- First lesson of first chapter in a subject: always unlocked.
- All other lessons: unlocked when previous lesson `completed=true` (ordered by chapter `order_index`, then lesson `order_index`).
- Chapter boundary: first lesson of any chapter (except the first) is locked until ALL lessons in the preceding chapter are `completed=true`.
- Chapter quiz: unlocked when all lessons in that chapter are `completed=true`.

### Chapter Completion Detection
After every lesson completion, `ProgressService`:
1. Gets `chapter_id` from the completed `lesson_id`
2. Counts total lessons in that chapter
3. Counts `lesson_progress` rows where `learner_id = ?` AND lesson belongs to chapter AND `completed = true`
4. If equal → chapter complete → trigger quiz generation background task (idempotent: skip if `chapter_quizzes` row already exists)

### Quiz Generation
- Triggered by chapter completion detection in `ProgressService`
- `QuizGenerationService` fetches learner's `lesson_progress` for all lessons in the chapter
- Computes average stars → sets quiz difficulty:
  - avg ≥ 2.5 → hard
  - avg ≥ 1.5 → medium
  - avg < 1.5 → easy
- Calls Claude with standards and performance context → generates personalized quiz JSONB (7 exercises minimum, mixed difficulty, weighted toward weak standards)
- Stores in `chapter_quizzes`
- Idempotent: if row exists, skip

### Streak Logic
On every lesson completion, `ProgressService` compares today's date (UTC) against `learners.last_active_at`:
- Same calendar day → no change to `streak_days`
- Yesterday → `streak_days += 1`
- Older → `streak_days = 1` (reset)

`last_active_at` updated to `now()` on every lesson completion.

### Learner Authentication
- Learners do not have their own accounts.
- All learner-facing requests use the parent's Cognito JWT + `learner_id` as a path parameter.
- `LearnerService` verifies `learner_id` belongs to the authenticated parent on every request.

### Parent PIN
- 4-digit PIN stored as bcrypt hash on `parents`.
- `POST /parent/verify-pin` returns a short-lived token (15-minute JWT) scoped to parent dashboard access.

### Content Generation (Standards Sync)
- On app startup: check if `standards` table is empty → if yes, fire non-blocking background sync.
- Sync fetches NY State standards from Common Standards/ASN API, inserts into `standards` (idempotent — skip existing).
- For each standard: call Claude → generate lesson JSONB with minimum 5 exercises, mixed exercise types, each exercise labeled with `difficulty: easy|medium|hard`.
- Lessons within a chapter generated with difficulty progressing easy → medium → hard by `order_index`.
- If sync is interrupted (app restart), it resumes from where it left off (idempotent inserts).

### Chapter Seeding
- `chapters` rows are seeded via Alembic migration (hand-crafted groupings by topic and grade level).
- Chapters are not generated dynamically — they define the curriculum structure that lessons populate.
- Standards fetched from the API are assigned to chapters during sync based on subject and grade.

### Answer Validation
- All exercise IDs present in lesson/quiz JSONB must be present in the submitted answers.
- Missing exercise → 422 Unprocessable Entity.
- Correct answers are never sent to the client — stripped from JSONB before every response.

### Leaderboard
- Global in v1 (all learners, all grade levels).
- Tie-breaking: `total_stars DESC`, then `streak_days DESC`, then `created_at ASC`.

## API Endpoints

### Auth
```
POST /auth/register       — parent registers via Cognito
POST /auth/login          — returns Cognito JWT
POST /auth/refresh        — refresh token
```

### Health
```
GET /health               — returns 200 OK
```

### Learners
```
GET    /learners                    — list parent's learners
POST   /learners                    — create learner profile
GET    /learners/{learner_id}       — learner detail (stars, level, xp, streak)
PATCH  /learners/{learner_id}       — update name, avatar, grade
```

### Curriculum
```
GET /subjects/{subject}/chapters?learner_id={id}
    — chapters + lessons with lock state and learner's stars per lesson + chapter quiz state

GET /lessons/{lesson_id}
    — sanitized lesson JSONB (correct answers stripped)
```

### Exercises (server-side grading)
```
POST /lessons/{lesson_id}/check-answer
     body: { exercise_id, answer }
     returns: { correct, explanation }

POST /chapter-quizzes/{quiz_id}/check-answer
     body: { exercise_id, answer }
     returns: { correct, explanation }
```

### Progress
```
POST /learners/{learner_id}/progress
     body: {
       lesson_id: uuid,
       time_seconds: int,
       answers: {
         "ex_1": { "selected_option_id": "b" },          // multiple_choice
         "ex_2": { "selected_word": "on" },               // fill_blank
         "ex_3": { "pairs": [["🐟 Fish", "🌊 Ocean"]] }  // matching
       }
     }
     returns: { stars_earned, correct, total, xp_earned, level_up }

GET  /learners/{learner_id}/progress
     returns: overall progress summary per subject

GET  /learners/{learner_id}/progress/{subject}
     returns: chapter + lesson breakdown with completion/stars per lesson
```

### Chapter Quizzes
```
GET  /chapter-quizzes/{quiz_id}
     returns: sanitized quiz JSONB (correct answers stripped)

POST /chapter-quizzes/{quiz_id}/submit
     body: {
       time_seconds: int,
       answers: { ... }   -- same format as lesson progress answers
     }
     returns: { stars_earned, effective_stars, xp_earned, level_up }
```

### Leaderboard
```
GET /leaderboard?period=weekly|monthly|all_time
    returns: [{ rank, name, avatar_emoji, total_stars, streak_days }]
```

### Parent Dashboard
```
POST /parent/verify-pin
     body: { pin }
     returns: { token }   -- 15-min parent JWT

GET  /parent/learners/{learner_id}/stats
     returns: {
       time_per_subject: { math, science, english },
       mastered: [{ lesson_id, title, subject }],
       needs_practice: [{ lesson_id, title, subject }],
       recent_activity: [{ lesson_id, title, stars_earned, completed_at }]
     }
```

## Implementation Phases

### Phase 1 — Foundation
**Goal:** Project skeleton, DB connection, auth working end-to-end.

- FastAPI project setup with 3-layer structure (`api/`, `services/`, `daos/`, `db/`, `schemas/`, `clients/`, `core/`)
- PostgreSQL connection via SQLAlchemy async engine
- Alembic configured, initial migration with all tables and indexes
- AWS Cognito integration: `POST /auth/register`, `POST /auth/login`, `POST /auth/refresh`
- JWT middleware: all protected routes validate Cognito JWT
- Health check endpoint: `GET /health`
- Docker + docker-compose for local dev
- Kubernetes manifests (Deployment, Service, ConfigMap)
- pytest configured with async test DB (separate schema, rolls back after each test)

**Tests:** auth flow integration tests, DB connection test, health check test.

---

### Phase 2 — Learners & Profiles
**Goal:** Parents can create and manage learner profiles.

- `POST /learners`, `GET /learners`, `GET /learners/{id}`, `PATCH /learners/{id}`
- `LearnerDAO`: insert, get by parent, get by id, update
- `LearnerService`: validation (age 5–8, grade 0–3), ownership check (learner must belong to requesting parent)
- Parent PIN: hash on register, `POST /parent/verify-pin` returns 15-min JWT

**Tests:** learner CRUD, ownership enforcement, PIN verification, age/grade validation.

---

### Phase 3 — Standards Sync & Content Generation
**Goal:** Standards seeded, lesson content generated via Claude, curriculum served with lock states.

- Alembic seed migration: `chapters` rows created (hand-crafted topic groupings for Math, Science, English across grade levels)
- `standards` table populated from Common Standards/ASN API
- `ContentGenerationService`:
  - On startup: check if `standards` table empty → fire background sync
  - Fetch NY State Math, Science, English standards from API → insert (idempotent)
  - Per standard: call Claude → generate lesson JSONB (15 exercises, mixed types, per-exercise difficulty)
  - Lessons within chapter ordered easy → medium → hard by `order_index`
- `GET /subjects/{subject}/chapters?learner_id={id}` — chapters + lessons + lock states + chapter quiz state
- `GET /lessons/{lesson_id}` — sanitized JSONB (correct answers stripped before response)
- `LessonDAO`: get chapters by subject, get lesson by id, count lessons in chapter
- `LessonService`: compute lock states from learner's `lesson_progress`

**Tests:** lock state computation (first lesson unlocked, others locked), sanitized lesson response (no correct answers), startup sync idempotent, chapter boundary locking.

---

### Phase 4 — Progress & Gamification
**Goal:** Learners complete lessons, earn stars/XP, level up. Chapter quizzes generated and served.

- `POST /lessons/{lesson_id}/check-answer` — stateless per-exercise grading (server checks JSONB, returns feedback)
- `POST /learners/{learner_id}/progress` — batch grading:
  - Validate all exercise IDs present
  - Grade answers against DB JSONB
  - Calculate stars from accuracy
  - Upsert `lesson_progress` (best score only)
  - Compute XP delta, update learner `total_stars`, `xp`, `level`, `streak_days`, `last_active_at`
  - Detect chapter completion → trigger `QuizGenerationService` background task
- `QuizGenerationService`:
  - Fetch learner's lesson_progress for chapter
  - Compute avg stars → set difficulty (≥2.5=hard, ≥1.5=medium, <1.5=easy)
  - Call Claude with performance context → generate personalized quiz JSONB (≥7 exercises)
  - Insert into `chapter_quizzes` (idempotent)
- `GET /chapter-quizzes/{quiz_id}` — sanitized quiz JSONB
- `POST /chapter-quizzes/{quiz_id}/check-answer` — stateless per-exercise grading
- `POST /chapter-quizzes/{quiz_id}/submit` — grade, apply star multiplier, update `total_stars`/`xp`/`level`
- `GET /learners/{learner_id}/progress` — overall per-subject summary
- `GET /learners/{learner_id}/progress/{subject}` — chapter + lesson breakdown

**Tests:** star calculation all accuracy bands, XP delta on replay, best-score-only upsert, negative delta no-op, streak same-day/yesterday/reset, level-up detection, chapter completion detection, quiz generation triggered once (idempotent), quiz multiplier tiers (all 4 bands), answer validation (all exercises required), quiz not-found 404 when not generated yet.

---

### Phase 5 — Leaderboard & Parent Dashboard
**Goal:** Leaderboard live. Parent dashboard stats served.

- `GET /leaderboard?period=weekly|monthly|all_time`
- `LeaderboardDAO`: ranked query by `total_stars DESC`, `streak_days DESC`, `created_at ASC`; period filter via `last_active_at`
- `GET /parent/learners/{learner_id}/stats`: time per subject, mastered (3★), needs_practice (≤1★ and completed), recent activity
- Parent JWT middleware for parent-scoped routes

**Tests:** leaderboard ranking correct, tie-breaking correct, period filter works, empty period returns empty list, parent routes reject Cognito JWT without parent scope.
